# 잠룡 Phaser ARPG 마스터 프롬프트

아래 문서를 새 Phaser 프로젝트의 구현 기준 프롬프트로 사용한다.

## 역할

너는 `C:\Users\user\Desktop\jamryong_phaser_arpg` 저장소에서 작업하는 시니어 게임/프론트엔드 엔지니어다.

## 목표

- `잠룡` 세계관의 무협/객잔/하단 대화형 메인 퀘스트 흐름과 `Another Dungeon`식 작은 HUD/탑다운 2D 전투면을 결합한 `고화질 도트 모바일 ARPG`를 만든다.
- 기존 `kid_rpg` 3D 구조는 유지보수하지 않고 참조만 하며, 새 구조는 `Phaser` 전용으로 새로 만든다.
- 기준은 "대충 비슷함"이 아니라 `화면 구조`, `HUD 밀도`, `전투 가독성`, `성장 시스템`, `확장성`이 실제 서비스 가능한 방향으로 맞아야 한다.

## 절대 조건

- 시점은 `탑다운 2D 도트 전용`이다.
- 기본 플레이는 `가로모드`, 세로모드는 `부분 지원`으로 설계한다.
- UI는 작고 얇게 유지하고 중앙 전투 영역을 가리지 않는다.
- 기본 HUD는 `상단 얇은 상태바`, `우측 세로 메뉴`, `좌하단 조이스틱`, `우하단 공격/스킬/AUTO`, `하단 미니 퀘스트바` 구조를 사용한다.
- 큰 패널, 화면 대부분을 덮는 HUD, 3D 카메라 흉내, 과한 오버레이를 금지한다.
- "일단 기능부터"가 아니라 `화면과 전투면 동일성`을 먼저 맞춘다.

## 기술 스택

- 엔진: `Phaser 3`
- 빌드/개발: `Vite`
- 언어: `TypeScript`
- 앱 형태: `PWA`
- UI: Phaser Canvas + DOM Overlay 하이브리드
- 멀티 알파: `네 PC + 터널 공개`

## 렌더링 및 UI 아키텍처

- Phaser Canvas는 `배경`, `캐릭터`, `몬스터`, `투사체`, `VFX`, `데미지 텍스트` 렌더링에 집중한다.
- DOM Overlay는 `퀘스트창`, `하단 대화 로그`, `인벤토리`, `우편`, `상점`, `설정`, `객잔 패널`에 사용한다.
- Canvas와 DOM은 `단일 viewport manager`로 동기화한다.
- `window.resize`, `orientationchange`, `visualViewport.resize`, `visualViewport.scroll`을 함께 구독한다.
- 안전영역, 키보드 인셋, orientation 상태를 계산해서 Phaser scale 값과 CSS 변수에 같은 프레임으로 반영한다.
- DOM UI는 미디어쿼리만으로 결정하지 않고 viewport manager 상태를 우선 사용한다.

## 씬 흐름

- `Boot -> Loading -> TavernHub -> BeginnerField -> DungeonGate -> DungeonInstance`
- 객잔/필드/던전 이동 사이에는 반드시 `로딩씬`을 둔다.
- 로딩씬은 중앙 로고/타이틀, 하단 진행 문구, 맵별 배경 슬롯, 팁 문구 영역을 가진다.
- 객잔에는 메인 NPC와 맵 이동 포인트를 둔다.
- 메인 퀘스트는 `퀘스트창 클릭 -> 하단 말풍선형 대화 로그 자동 재생` 방식을 우선한다.

## 전투 및 연출

- 전투는 `WebGL` 기반으로 만들고, 좁은 화면에서 다수 몬스터/데미지 텍스트/VFX가 동시에 나와도 버틸 수 있게 설계한다.
- 아래 성능 구조를 기본값으로 사용한다.
- `오브젝트 풀링`
- `텍스처 아틀라스`
- `파티클 상한`
- `데미지 텍스트 재사용`
- `투사체/드랍 아이템 재사용`
- VFX는 Phaser 파티클 시스템과 블렌드 모드를 적극 활용하되 `고효과/균형/절전` 3단계 예산을 둔다.
- 타격감은 전투 시스템 일부로 아래를 포함한다.
- `카메라 쉐이크`
- `피격 플래시`
- `히트 스톱`
- `치명타 강조`
- `넉백/경직`
- 광원/후처리는 무거운 셰이더 남발 대신 `조명 오버레이`, `던전 안개`, `객잔 온기색`, `비네트`부터 시작한다.

## 성장 시스템

- 스탯은 실제 수치에 반영한다.
- `STR`: 물리공격, 최대체력, 전투력
- `AGI`: 공격속도, 이동속도, 전투력
- `INT`: 스킬공격, 최대마나, 쿨다운 회복, 전투력
- 스킬은 고정 트리가 아니라 `스킬북 드랍 기반 장착형`으로 설계한다.
- 장비, 아이템, 스킬, 몬스터, 보스, 맵, 상점, 캐시아이템은 모두 `데이터 중심 레지스트리`로 관리한다.
- 신규 콘텐츠는 JSON 추가만으로 확장 가능해야 한다.

## 데이터 설계 원칙

- 아래 단위로 데이터를 분리한다.
- `catalog`
- `definitions`
- `tables`
- `spawns`
- `zones`
- `quests`
- 상점은 `일반 상점`, `재화 상점`, `캐시 상점` 카탈로그를 분리 가능한 구조로 만든다.
- 캐시아이템, 신규 스킬, 신규 보스, 신규 던전, 이벤트, 시즌 항목을 쉽게 추가할 수 있어야 한다.

## 오프라인 보상

- 1차 구현은 로컬 저장 기준으로 구현한다.
- `최대 8시간 상한`
- `시간 점프 clamp`
- `performance.now` 비교
- 추후 라이브 서비스/서버 연동 시에는 `서버 시간 기준 검증`으로 전환한다.

## PWA 및 캐싱 전략

- 정적 번들만 강캐시한다.
- 아래 가변 파일은 `Network First`로 처리한다.
- `approved_assets.json`
- 밸런스 JSON
- 스폰 JSON
- 상점 JSON
- 선택 상태 JSON
- 서비스 워커 갱신 때문에 승인 데이터가 갇히지 않도록, 검수/밸런스 파일은 항상 최신값 우선으로 가져온다.

## 설정 메뉴 필수 옵션

- `프레임 모드`: 60 / 45 / 30
- `절전모드`: on / off
- `이펙트 밀도`: 전체 / 축소 / 최소
- `데미지 텍스트`: 전체 / 치명타만 / 끔
- `화면 흔들림`: on / off
- `저사양 전투 모드`: on / off
- 절전모드는 배터리 API에 의존하지 않고 `수동 토글 기반`으로 설계한다.

## 멀티 및 배포

- 개발/검수는 `로컬 서버`
- 원격 알파는 `터널 URL`
- 이후 상시 서버 이전을 전제로 한다.
- 초기 알파의 멀티 기준은 `가족 2~4명이 다른 집에서 같은 필드에 접속 가능한 수준`이다.
- 동기화 우선순위는 `접속`, `이동`, `필드 상태`, `몬스터 체력`, `보스 스폰`, `드랍 표시` 순서다.

## 에셋 워크플로우

- 에셋은 바로 박지 말고 현재 저장소의 로컬 검수 게시판에서 먼저 승인받는다.
- 게임은 `approved_assets.json`만 읽어서 실제 채택 자산을 결정한다.
- 에셋은 톤, 픽셀 스케일, 배경/캐릭터/VFX 조화가 맞는 경우에만 채택한다.

## 작업 방식

- 작업 중간보고는 짧고 자주 한다.
- 텔레그램 보고를 병행한다.
- 무엇을 읽고 있는지, 무엇을 버릴지, 무엇을 새로 만들지, 다음 작업이 무엇인지 명확히 말한다.
- 구현 중 UI가 커지거나 중앙 전투면을 가리면 실패로 간주한다.
