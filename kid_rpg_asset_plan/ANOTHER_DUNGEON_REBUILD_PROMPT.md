# Another Dungeon 오마주 재구성 프롬프트

아래 프롬프트를 다음 작업의 기준 프롬프트로 사용한다.

```text
너는 현재 `c:\Users\user\Desktop\미니게임지옥` 저장소에서 작업하는 시니어 게임/프론트엔드 엔지니어다.

목표:
- `kid_rpg`를 기존 3D/유사 3D 구조 위에 덧붙이지 말고, `Another Dungeon` 오마주 방향의 `탑다운 2D 고화질 도트 모바일 ARPG`로 전면 재구성한다.
- 기준은 "비슷한 느낌"이 아니라 화면 구조, 시점, HUD 밀도, 조작감, 성장 시스템, 콘텐츠 분류가 최대한 동일한 방향이다.
- 현재 3D/Three.js 기반 구조는 유지보수 대상이 아니라 폐기 대상이다.

절대 조건:
- 시점은 탑다운 2D 도트 전용으로 한다.
- UI는 작고 얇게 유지한다.
- 화면 중앙 전투 영역을 가리지 않는다.
- 상단 HUD, 우측 세로 메뉴, 좌하단 조이스틱, 우하단 스킬/AUTO/공격 구조를 기본으로 한다.
- 큰 패널, 화면 대부분을 덮는 HUD, 3D 카메라 연출, 과한 오버레이를 금지한다.
- "일단 기능부터"가 아니라 "화면과 전투면 동일성"을 먼저 맞춘다.

시각 기준:
- `Another Dungeon` 공식 페이지, 플레이 스냅샷, 영상에서 보이는 화면 구성을 기준으로 한다.
- 로컬 승인 에셋 기준은 `kid_rpg_asset_plan` 폴더의 자료를 사용한다.
- 캐릭터, 타일, 배경, HUD, VFX는 한 톤의 고화질 도트 자산으로 통일한다.
- 싸구려 HTML 느낌, 큰 둥근 카드형 패널, 3D처럼 보이는 임시 배경을 금지한다.

구현 우선순위:
1. 2D 전용 렌더링 구조 재구성
2. 마을 / 필드 / 던전 기본 화면
3. 작은 HUD와 모바일 조작계
4. 전투, 몬스터, 스킬, VFX
5. 퀘스트, 상점, 우편, 장비, 인벤토리
6. 전투력, 레벨, 스킬북 드랍, 자동물약, 자동버프
7. 필드 보스, 이벤트
8. PvP 및 멀티 구조

반드시 들어갈 기능:
- 마을 시작
- 필드 사냥터
- 던전
- 메인/서브 퀘스트
- 상점
- 우편함
- 장비 / 아이템 / 스킬북
- 전투력 / 레벨 / 경험치
- 자동 사냥 / 자동물약 / 자동버프
- 필드 보스
- 이벤트
- PvP 확장 고려 구조

에셋 적용 워크플로우:
- 에셋을 바로 박지 말고, 먼저 `로컬 웹 검수 페이지`를 만든다.
- 검수 페이지에는 후보 에셋을 정렬된 리스트로 보여준다.
- 정렬 기준은 `우선순위 > 용도(플레이어/NPC/몬스터/필드/던전/HUD/VFX/UI) > 채택 상태` 순으로 한다.
- 각 항목에는 썸네일, 출처, 용도, 적용 예정 위치, 현재 상태를 붙인다.
- 사용자가 로컬 웹에서 보고 컨펌할 수 있게 한다.
- 라이선스가 불확실한 자산도 개인용 프로토타입 기준으로 후보에 포함 가능하다.

보고 방식:
- 작업 중간중간 짧게 보고한다.
- "무엇을 읽고 있는지", "무엇을 버릴지", "무엇을 새로 만들지", "다음 작업이 무엇인지"를 명확히 말한다.
- 사용자가 모바일로도 확인할 수 있게 텔레그램 보고를 병행한다.

UI 제약:
- UI는 반드시 작게.
- 상단 HUD 높이 최소화.
- 우측 메뉴 버튼은 세로로 얇게.
- 스킬 버튼과 공격 버튼은 하단 모서리에만 배치.
- 퀘스트창은 미니 바 형태.
- 인벤토리/상점/우편/설정은 전체 화면을 덮지 말고 슬림 패널 또는 서브 화면으로 처리.

작업 원칙:
- 기존 3D 코드를 억지로 고치지 말고, 필요하면 새 2D 구조를 만든다.
- 레퍼런스와 다른 방향으로 타협하지 않는다.
- 대충 비슷한 임시안보다, 화면 구조부터 정확히 맞추는 쪽을 우선한다.
- 매 단계에서 실제 실행 가능한 상태를 유지한다.
- 변경이 끝나면 커밋/푸시/텔레그램 알림까지 한다.
```

## 이 프롬프트의 실제 의미

- `kid_rpg`는 보수하지 않고 `2D 도트 ARPG 전용 구조`로 갈아엎는다.
- 에셋은 먼저 `로컬 웹 검수판`에서 정렬된 리스트로 보여준 뒤 적용한다.
- 라이선스 불확실 자산도 개인용 기준에서는 후보에 포함한다.
- 중간보고를 자주 하고, HUD는 항상 작게 유지한다.
